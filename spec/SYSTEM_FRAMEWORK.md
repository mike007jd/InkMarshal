# SYSTEM FRAMEWORK - InkMarshal

Last checked: 2026-07-18

This document is the maintained system map for the current codebase. It should describe what exists now, not historical plans.

## Product Boundary

InkMarshal is a local-first desktop writing Studio.

- Desktop is the product: Tauri v2 embeds the Next.js standalone app, local SQLite, OS keychain access, and local model engine control.
- The public website is a separate build in sibling repository `../AiNovelSite`; it owns landing, examples, privacy, terms, `/studio`, and `/download` and does not host a writing workspace.
- There is no login, cloud account, cloud database, platform credit balance, or server-owned provider fallback.
- Launch scope is macOS Apple Silicon. Windows is not release-ready and must not get a public download URL until signing and validation exist.

## Runtime Shape

```text
Tauri desktop app
  -> Next.js standalone server
  -> local SQLite at app_data_dir/inkmarshal.db
  -> Rust capability layer
  -> bundled or detected local model runtime
  -> optional BYOK provider connection from local settings/keychain

Public web deployment (`../AiNovelSite`)
  -> static/SSR landing and download routes
  -> no Studio data access
  -> no user runtime
```

`proxy.ts` enforces the boundary:

- Production web returns 404 for `/api/*` except `/api/health`.
- An accidental production Web runtime redirects desktop-only routes to `https://www.inkmarshal.com/download`.
- Desktop sessions use the Tauri-provided token/header/cookie path.

## Data Model

The only shipped data mode is local.

- Database: `better-sqlite3`, file stored under the Tauri app data directory.
- User: fixed local user shim in `lib/local-auth.ts`.
- Secrets: BYOK credentials stored through `src-tauri/src/secret.rs`, not environment variables.
- Vault: optional markdown vault sync is file-backed and validated through the Rust filesystem bridge plus `lib/vault/*`.
- Web runtime: no database access for product data.

Schema and migration ownership lives in:

- `lib/db/schema/*`
- `lib/db/migrations.ts`
- `lib/db/queries-*`

## AI And Model Supply

All generation must route through the model-supply layer before touching a provider.

```text
operation
  -> capability role
  -> configured connection
  -> provider factory
  -> AI SDK call
```

Primary ownership:

- `lib/model-supply/*`: operation, capability, connection, readiness, local-engine registration.
- `lib/ai-providers.ts`: provider factory.
- `lib/providers.ts`: current BYOK provider catalog and source-backed model ids.
- `lib/ai/*`: prompt runners, structured outputs, chapter quality, editing, unification.
- `lib/ai-usage.ts`: local usage accounting and stream lifecycle.

Do not add server-owned provider API keys, hidden env fallbacks, or platform credit accounting.

## Writing Pipeline

The long-form writing flow is:

```text
new novel
  -> discovery interview
  -> greenlight pack
  -> blueprint projection from outline knowledge entries
  -> autonomous chapter writing
  -> chapter memory and quality checks
  -> manuscript editing/rewrite
  -> whole-book unification
  -> export
```

Important boundaries:

- Outline knowledge entries are the blueprint source of truth.
- `start-writing` owns autonomous drafting and writing locks.
- Chapter continue/rewrite/edit routes are non-chat writing workflows and may use project NDJSON frames.
- Chat routes use AI SDK UIMessage streams, not a project-owned partial-message protocol.
- Whole-book unification stores structured edits and applies them with chapter version checks.

## Streaming Contracts

There are two allowed stream classes:

1. Chat: AI SDK v6 UIMessage streams through `streamText(...).toUIMessageStreamResponse()`.
2. Non-chat writing/editor workflows: project NDJSON frames from `lib/streaming-helpers.ts` when workflow state must travel with the stream.

No `/messages/partial` endpoint should exist.

## Desktop Capability Layer

Rust owns native capabilities that browser code cannot safely perform:

- Engine lifecycle and health.
- Model search, download, resume, and sha256 verification.
- Keychain secret storage.
- Vault file access.
- Menu events and app metadata.

Tauri permissions are intentionally scoped to the app-owned frontend origin, with validation in command bodies for high-impact actions.

## Public Web Layer (`../AiNovelSite`)

The public web app owns:

- Landing and product positioning.
- Download handoff.
- Example library.
- Privacy and terms.
- SEO, sitemap, robots, Open Graph image.
- Optional Vercel Analytics and Speed Insights on web only.

It must not expose a hosted Studio, login shell, provider key entry, model console, or generation API.

## Release Shape

The release path is macOS-only:

- `pnpm release:mac` builds signed and notarized macOS assets.
- Stable asset name: `InkMarshal-mac-aarch64.dmg`.
- Canonical URL: `https://github.com/mike007jd/InkMarshal/releases/latest/download/InkMarshal-mac-aarch64.dmg`.
- `pnpm verify:release-desktop` validates the desktop bundle, updater configuration, signatures, and published release assets. `../AiNovelSite` independently validates canonical site/download URLs during Vercel Production builds.

Detailed release steps live in `novelcraft-ai/docs/LAUNCH_READINESS.md`.
